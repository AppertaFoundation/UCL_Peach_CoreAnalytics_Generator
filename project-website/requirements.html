Project Background

Currently, the NHS does not provide a suitable infrastructure to be used for activities such as capturing, transforming or transferring data across different systems, where users can use their own encrypted data flows as well as manipulate the incoming data where necessary. Because of this lack of a single secure, consistent and easy to use platform, existing applications for healthcare services are not widely used.

The vision for Project PEACH is to create such an infrastructure to support research, education and practice progression of NHS Systems for different healthcare related cases in Radiology.

Our Client


Our main client is Dr Navin Ramachandran who is a Radiology Doctor at UCLH and an honorary senior lecturer at UCL CHIME. He will oversee the PEACH project.

Problem Statement

Our main problem is to implement the core infrastructure of the system such that various applications can connect to the servers from various domains, building upon the largely experimental work carried out by the team previously allocated to this project.

Our secondary problem is to provide researchers and clinicians with a random data generator where the tool must be able to generate new anonymized sets of data and thus enable researchers and clinicians to use this data for system building and testing while preserving data confidentiality.

Requirement Gathering

Our main source of requirements were Interviews and meetings with Dr Ramachandran. After our initial meeting, we started working on a set of MoSCoW style analysis and created our initial set of requirements. Few weeks later, after a broader understanding of our project and its scope, we came up with the final set of requirements keeping feasibility in mind which both us and our client agreed upon.

Since the Generator is a continuation of a previous team's project, we thoroughly looked at their requirements and analysed what was complete and incomplete. Based on that, we and Dr Ramachandran came up with our version of requirements.

Requirement Overview

Core Analytics

  • The PEACH infrastructure must be able to communicate with applications from various domains
  • There must be a director of data flow to control the input, output and pre processing of the data
  • There must be a central message exchange hub to direct the data inside the container service
  • The processed data should be persisted as logs

 

PEACH Generator

  • The generator must be able to specify the type of output of the data
  • The generator must allow the user to choose the privacy level of certain data fields
  • The generator must implement the k-anonymity anonymisation model.
  • The generator must be able to shuffle existing data
  • The generator must be able to generate new randomized data from previous novel data sets
